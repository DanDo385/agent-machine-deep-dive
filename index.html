Agent Machine

Strip away the branding, the feature lists, the GitHub star counts. Every agent — Hermes, OpenClaw, Claude Code, AutoGPT, LangChain agents, your custom bot — is the same machine with the same five parts. The features are trim. The skeleton is this:

This document does a deep dive on each part: what it is, how it works, real-world analogies, how to evaluate it, code examples, and exactly what breaks when it fails — and how to fix it.

How the Parts Move Together

Think of a 911 dispatch center. A call comes in (Platform Input). The dispatcher pulls up the caller's address and history (Session Hydration). The dispatcher thinks through the situation (Agent Loop). They send police/fire/ambulance (Tool Dispatch). The outcome gets logged in the case file (Persistence). Next time the same address calls, the dispatcher sees the history immediately.

The feedback loop is what makes it an "agent" rather than a chatbot. A chatbot fires once and stops. An agent loops: the result of tool dispatch feeds back into the agent loop, which may trigger more tool dispatch, which feeds back again, until the model decides it's done. This is the difference between "tell me about the weather" and "check the weather, compare it to last week, write a summary, and email it to me."

Each part has a job. When all five do their jobs cleanly, the system feels magical. When one fails, the others can't compensate — the failure propagates forward through the chain.

PART 1

Platform Input — The Ears

What It Is

Platform Input is the layer that converts raw platform events (a Discord message, a Telegram voice note, an HTTP POST, a CLI keystroke, a cron tick) into a normalized, clean message object the rest of the system can reason about. Its only job is translation. It doesn't think. It doesn't store. It listens and converts.

Think of it like a simultaneous interpreter at the UN. The speaker says something in Arabic. The interpreter doesn't have opinions about what was said. They translate it into English and hand it to the next person. Platform Input is that interpreter — except instead of languages, it's converting platform-native event formats (Discord's WebSocket payloads, Telegram's Bot API JSON, raw stdin bytes) into one universal message format your agent understands.

How It Works

Every platform speaks a different dialect. Discord sends an event payload with guild_id, channel_id, author, content, attachments. Telegram sends update_id, message.from.id, message.text. A CLI sends raw bytes from stdin. Platform Input normalizes all of these into one canonical struct:

type IncomingMessage struct {
    // Who sent it
    UserID      string
    UserHandle  string

    // Where it came from
    Platform    string    // "discord", "telegram", "cli"
    ChannelID   string    // the routing key for session lookup

    // What they said
    Text        string
    Attachments []Attachment

    // Metadata
    Timestamp   time.Time
    MessageID   string    // platform-native ID (for threading, replies)
    IsAuthorized bool    // did this user pass the allowlist check?
}

// Each platform adapter implements this interface
type PlatformAdapter interface {
    Start(ctx context.Context, out chan<IncomingMessage) error
    Send(channelID string, text string) error
    SendTyping(channelID string) error
}
// Discord adapter — converts Discord event into IncomingMessage
func (a *DiscordAdapter) onMessageCreate(s *discordgo.Session, m *discordgo.MessageCreate) {
    if m.Author.Bot { return } // ignore other bots

    msg := IncomingMessage{
        UserID:      m.Author.ID,
        UserHandle:  m.Author.Username,
        Platform:    "discord",
        ChannelID:   m.ChannelID,        // key line — routing key
        Text:        m.Content,
        Timestamp:   time.Now(),
        MessageID:   m.ID,
        IsAuthorized: a.allowlist.Contains(m.Author.ID),
    }

    // Handle attachments (images, files)
    for _, att := range m.Attachments {
        msg.Attachments = append(msg.Attachments, Attachment{
            URL:      att.URL,
            Filename: att.Filename,
            MimeType: att.ContentType,
        })
    }

    a.out <- msg  // send into the pipeline
}
Hermes gateway has 20 platform adapters doing exactly this. Your OpenClaw Discord bot is a single-platform version of the same pattern. The adapter for Signal looks almost identical to the one for Discord — just different library calls. The normalized IncomingMessage struct is the same in both cases, which is why you can add a new platform without touching the agent loop.

Authorization Lives Here

Platform Input is also the security perimeter. It should check authorization before the message ever reaches the agent loop. Fail fast, fail early. If a user isn't on the allowlist, drop the message here — don't waste a token on it.

Evaluating Platform Input

Working Well
Every platform event arrives as a clean, normalized struct. The agent loop never sees platform-specific fields. Authorization is checked before any tokens are spent. Attachments are pre-fetched or at least pre-identified. Latency from event to pipeline is under 100ms.
Breaking Down
The agent loop contains if-statements like if platform == "discord". Unauthorized users reach the LLM. Attachments arrive as raw URLs with no MIME type info. Events are dropped silently. The same message gets processed twice (Discord resend / Telegram retry quirk).
The agent loop should be completely unaware of which platform sent the message. If you can swap Discord for Telegram without touching agent logic, Platform Input is working correctly.

Individual Failure Example: Duplicate Message Processing

// you get the same update twice. This fires the agent twice.

}

✓ FIXED — Idempotency key deduplication
type InputLayer struct {
    seen *sync.Map  // message_id → processed time
}

func (il *InputLayer) process(update tgbotapi.Update) {
    msgID := fmt.Sprintf("tg:%d", update.Message.MessageID)

    // Check if already processed
    if _, exists := il.seen.Load(msgID); exists {
        return  // already handled, skip
    }

    // Mark as seen immediately (before processing, not after)
    il.seen.Store(msgID, time.Now())

    msg := normalize(update)
    pipeline <- msg
}

// Periodically clean old entries (older than 1 hour)
func (il *InputLayer) cleanup() {
    il.seen.Range(func(k, v interface{}) bool {
        if time.Since(v.(time.Time)) > time.Hour {
            il.seen.Delete(k)
        }
        return true
    })
}
PART 2

Session Hydration — The Memory Recall

What It Is

Session Hydration is the act of loading everything the agent needs to know before it starts thinking. The LLM has no persistent memory between API calls. Every single call to the API is a blank slate. Session Hydration is the process of filling in that blank slate — loading conversation history, user memory, system prompt components, skill files, and any other context the agent needs.

Imagine a doctor seeing patients. Between appointments, the doctor remembers nothing — complete amnesia. Before each appointment, a nurse hands the doctor a briefing folder: "This is Dan. Last visit was 3 weeks ago. He has hypertension. His main concern today is X. He mentioned Y last time." The doctor reads the folder, walks in, and picks up exactly where things left off. Session Hydration is that briefing folder. Without it, every conversation starts from scratch: "Hi, I'm an AI assistant. How can I help you today?"

How It Works

Session Hydration assembles the messages array that gets sent to the LLM API. This is a list of alternating user/assistant turns. It also assembles the system prompt. Together these are the complete "memory" the LLM has access to for this inference call.

type SessionHydrator struct {
    db          *SessionDB
    memoryStore *MemoryStore
    skillStore  *SkillStore
    maxTokens   int
}

func (h *SessionHydrator) Hydrate(msg IncomingMessage) (HydratedSession, error) {
    // 1. Load conversation history for this channel
    history, _ := h.db.LoadHistory(msg.ChannelID, 50) // last 50 turns

    // 2. Load long-term memory for this user
    memory, _ := h.memoryStore.LoadUserMemory(msg.UserID)

    // 3. Load relevant skills
    skills, _ := h.skillStore.GetActiveSkills(msg.ChannelID)

    // 4. Build system prompt by assembling components
    systemPrompt := h.buildSystemPrompt(memory, skills)

    // 5. Budget tokens — don't overflow the context window
    messages := h.budgetHistory(history, h.maxTokens - h.estimate(systemPrompt))

    // 6. Append the incoming message
    messages = append(messages, Message{Role: "user", Content: msg.Text})

    return HydratedSession{
        SystemPrompt: systemPrompt,
        Messages:     messages,
        SessionKey:   msg.ChannelID,
        UserID:       msg.UserID,
    }, nil
}

func (h *SessionHydrator) buildSystemPrompt(memory UserMemory, skills []Skill) string {
    var parts []string

    // Core personality — soul.md equivalent
    parts = append(parts, h.loadSoul())

    // Long-term memory about this user
    if memory.Summary != "" {
        parts = append(parts, "## User Context\n" + memory.Summary)
    }

    // Active skills for this session
    for _, skill := range skills {
        parts = append(parts, skill.Content)
    }

    return strings.Join(parts, "\n\n")
}

// Token budgeting — trim oldest messages first to fit context window
func (h *SessionHydrator) budgetHistory(history []Message, budget int) []Message {
    total := 0
    result := []Message{}

    // Walk backwards from most recent
    for i := len(history) - 1; i >= 0; i-- {
        tokens := h.estimate(history[i].Content)
        if total + tokens > budget { break }
        total += tokens
        result = append([]Message{history[i]}, result...) // prepend
    }
    return result
}
Your OpenClaw setup: When Dex receives a Discord message in the #sandbox channel, Session Hydration is what loads soul.md, the sandbox-memory.md file, the relevant skill files for that channel, and the last N turns of conversation history — all before the LLM API call fires. The per-channel memory design you chose is exactly the right isolation boundary.

The Token Budget Problem

This is Session Hydration's hardest problem. Models have a finite context window. Your history + system prompt + skills + new message must fit inside it. If you naively load everything, you'll blow the window and get an API error. The strategies in ascending sophistication:

StrategyHowCost
Truncate oldestKeep only the last N messagesLoses old context silently
Summarize middleLLM call to compress old turnsExtra API cost, some fidelity loss
Semantic recallVector search for relevant past messagesComplex infrastructure (embeddings DB)
Tiered memoryHot (recent) + warm (summary) + cold (search)Highest complexity, best results

Evaluating Session Hydration

Working Well
Agent references things said in prior sessions accurately. System prompt is stable across turns (not rebuilt from scratch each message). Token count is tracked and never exceeds context limit. History load is fast (<50ms typically from SQLite). Skills relevant to the channel are loaded; irrelevant ones are not.
Breaking Down
Agent says "I don't have information about previous conversations" despite having history. System prompt is rebuilt differently each message causing prompt cache misses. History load causes a 2+ second delay. Loading ALL skills regardless of channel (wastes tokens). Context overflow crashes the API call.

Individual Failure Example: No Token Budget = Context Overflow

✗ BAD — Loads full history, no token limit
func hydrateSession(channelID string) []Message {
    // Load ALL messages for this channel — no limit
    rows, _ := db.Query("SELECT role, content FROM messages WHERE channel=? ORDER BY ts ASC", channelID)
    var messages []Message
    for rows.Next() {
        var m Message
        rows.Scan(&m.Role, &m.Content)
        messages = append(messages, m)
    }
    return messages  // DISASTER after ~2 weeks of chat
}
// After 3 weeks of use: 8,000 messages × ~100 tokens = 800k tokens
// Context window: 200k tokens. API returns: "context_length_exceeded"
✓ FIXED — Token-budgeted history with compression fallback
const MAX_HISTORY_TOKENS = 80_000  // ~40% of 200k window

func hydrateSession(channelID string, systemPromptTokens int) []Message {
    budget := MAX_HISTORY_TOKENS - systemPromptTokens

    // Load recent messages with token counting
    rows, _ := db.Query(`
        SELECT role, content, token_count
        FROM messages
        WHERE channel=?
        ORDER BY ts DESC
        LIMIT 200
    `, channelID)

    var recent []Message
    totalTokens := 0

    for rows.Next() {
        var m Message
        var tokens int
        rows.Scan(&m.Role, &m.Content, &tokens)

        if totalTokens + tokens > budget { break }
        totalTokens += tokens
        recent = append(recent, m)
    }

    // Reverse to chronological order
    for i, j := 0, len(recent)-1; i < j; i, j = i+1, j-1 {
        recent[i], recent[j] = recent[j], recent[i]
    }

    // Optional: prepend a compression summary if history was cut
    if rows.Next() {  // more rows exist that we cut off
        summary := db.GetSummary(channelID)  // pre-computed nightly
        if summary != "" {
            recent = append([]Message{{
                Role: "system",
                Content: "[Earlier context summary: " + summary + "]",
            }}, recent...)
        }
    }

    return recent
}
PART 3

Agent Loop — The Brain

What It Is

The Agent Loop is the core reasoning engine. It's a while loop that calls the LLM API, reads the response, and decides what to do next. It does NOT execute tool calls itself — it hands those to Tool Dispatch. It does NOT store anything — it hands that to Persistence. Its only job is to think: call the LLM, interpret the output, and orchestrate what happens next.

The Agent Loop is a chess player in a tournament. The player looks at the board (the messages array), thinks about the position (the LLM call), makes a move (decides to use a tool OR give a final answer), then waits to see what happens (tool result or user response), and looks at the board again. The player doesn't physically move the pieces — a tournament official does (Tool Dispatch). The player doesn't record the game — a scorekeeper does (Persistence). The player just thinks and decides.

How It Works

The LLM has two possible responses: "I'm done" (stop_reason = end_turn) or "I need to do something" (stop_reason = tool_use). The loop handles both:

type AgentLoop struct {
    client      LLMClient
    dispatcher  ToolDispatcher
    maxTurns    int     // safety: stop runaway loops
}

func (al *AgentLoop) Run(ctx context.Context, session HydratedSession) (AgentResult, error) {
    messages := session.Messages
    turns := 0

    for {
        turns++
        if turns > al.maxTurns {
            return AgentResult{}, fmt.Errorf("max turns (%d) exceeded", al.maxTurns)
        }

        // ── THE CORE LLM CALL ──────────────────────────────────────────
        response, err := al.client.Complete(ctx, CompletionRequest{
            System:   session.SystemPrompt,
            Messages: messages,
            Tools:    al.getAvailableTools(),
            Model:    "claude-sonnet-4-20250514",
        })
        if err != nil { return AgentResult{}, err }

        // ── INTERPRET THE RESPONSE ─────────────────────────────────────
        switch response.StopReason {

        case "end_turn":
            // Model is done — extract final text and return
            finalText := response.GetText()
            return AgentResult{
                Text:     finalText,
                Messages: messages,
                Turns:    turns,
            }, nil

        case "tool_use":
            // Model wants to use tools — hand off to dispatcher

            // 1. Append the assistant's message (with tool_use blocks)
            messages = append(messages, Message{
                Role:    "assistant",
                Content: response.Content,  // includes tool_use blocks
            })

            // 2. Execute ALL tool calls in this response
            toolResults, err := al.dispatcher.Execute(ctx, response.ToolCalls)
            if err != nil { return AgentResult{}, err }

            // 3. Append tool results as a user message
            messages = append(messages, Message{
                Role:    "user",
                Content: toolResults,  // tool_result blocks
            })

            // 4. Loop back — model will see results and decide next action
            continue

        default:
            return AgentResult{}, fmt.Errorf("unexpected stop reason: %s", response.StopReason)
        }
    }
}

The messages Array — The Actual Working Memory

This is the most important data structure in the entire system. As the loop runs, the messages array grows. Here's what it looks like mid-task:

// What the messages array looks like during a multi-step task:
// "Research AAPL and write a summary"

messages = [
  {"role": "user",      "content": "Research AAPL and write a summary"},
  {"role": "assistant", "content": [{"type": "tool_use", "name": "web_search", "input": {"query": "AAPL Q2 2026 earnings"}}]},
  {"role": "user",      "content": [{"type": "tool_result", "content": "Apple reported $91B revenue..."}]},
  {"role": "assistant", "content": [{"type": "tool_use", "name": "web_search", "input": {"query": "AAPL analyst ratings 2026"}}]},
  {"role": "user",      "content": [{"type": "tool_result", "content": "Goldman Sachs Buy $230..."}]},
  {"role": "assistant", "content": "# Apple Inc. Summary\n\nApple reported..."}  // final answer
]

// The model sees ALL of this on every loop iteration.
// It can "remember" what the first search returned because it's in the array.
// This is not magic — it's just text passed to an API.
Real case — Dex market update cron: The agent loop runs, calls web_search for BTC price, then calls it again for ETH, then calls write_file to save the result to the vault, then returns a formatted message. That's 4 loop iterations — 3 tool_use rounds plus 1 end_turn. The messages array at the end has 8 entries (1 user + 3 assistant tool_use + 3 tool_results + 1 assistant final). All that happens inside one call to AgentLoop.Run().

Safety: The maxTurns Guard

Without a loop limit, a buggy tool or confused model can loop forever — burning tokens and money. Always have a max turns ceiling. A reasonable default is 20 for interactive agents, 10 for cron jobs.

Evaluating the Agent Loop

Working Well
Model consistently uses the right tools for the right tasks. Multi-step tasks complete without human intervention. Loop terminates cleanly (end_turn, not timeout). Tool use is purposeful — no redundant search calls. Turn count for typical tasks is predictable and low (2-5 turns for most tasks).
Breaking Down
Model calls the same tool repeatedly with the same arguments (stuck in a loop). Tool results are being ignored (model doesn't update its plan). Max turns is hit on simple tasks. Model returns "end_turn" without actually completing the task. Model hallucinates tool names that don't exist in the registry.

Individual Failure Example: Infinite Tool Loop

✗ BAD — Model loops on a broken tool, no circuit breaker
// web_search returns an error. The model sees "error" and tries again.
// The tool keeps erroring. The model keeps trying. No max turns.
// This runs until you hit API rate limits or your wallet is empty.

for {
    resp, _ := llm.Complete(messages, tools)
    if resp.StopReason == "end_turn" { return resp }
    results := dispatcher.Execute(resp.ToolCalls) // keeps erroring
    messages = append(messages, resp, results)
    // loop forever ↑
}
✓ FIXED — maxTurns + consecutive error detection
turns := 0
consecutiveErrors := 0
maxTurns := 20
maxConsecutiveErrors := 3

for {
    turns++
    if turns > maxTurns {
        return fmt.Errorf("safety limit: exceeded %d turns", maxTurns)
    }

    resp, _ := llm.Complete(messages, tools)
    if resp.StopReason == "end_turn" { return resp }

    results, err := dispatcher.Execute(resp.ToolCalls)

    if err != nil {
        consecutiveErrors++
        if consecutiveErrors >= maxConsecutiveErrors {
            // Tell the model the tool is broken, let it adapt
            messages = append(messages, SystemMessage{
                Content: "Tool unavailable. Please complete the task with available information.",
            })
            consecutiveErrors = 0
        }
    } else {
        consecutiveErrors = 0  // reset on success
    }

    messages = append(messages, resp, results)
}
PART 4

Tool Dispatch — The Hands

What It Is

Tool Dispatch is the execution layer. When the agent loop says "call the web_search tool with query=AAPL earnings", Tool Dispatch is the system that actually goes and does that — makes the HTTP call, runs the bash command, writes the file, sends the API request. It receives a tool call specification from the agent loop, executes it in the real world, and returns the result.

Tool Dispatch is the hands of a surgeon, responding to the surgeon's brain. The brain (Agent Loop) thinks "cut here, 3 centimeters". The hands (Tool Dispatch) pick up the scalpel and actually make the cut. The hands don't think — they execute with precision. But if the hands are trembling (slow tool execution), or they pick up the wrong instrument (wrong tool called), or the scalpel is dull (buggy tool implementation), the whole operation fails — even if the brain was right.

The Tool Registry Pattern

Every agent uses a registry. Tools self-register at startup. The dispatcher looks up tools by name and executes them. This is clean dependency injection:

// tools/registry.go — no dependencies, imported by all tools
type ToolFn func(ctx context.Context, args map[string]interface{}) (string, error)

type ToolDef struct {
    Name        string
    Description string
    Schema      JSONSchema  // what args the tool accepts
    Fn          ToolFn
    Dangerous   bool       // requires human approval?
    Tags        []string   // for filtering by context
}

var globalRegistry = map[string]*ToolDef{}

func Register(def ToolDef) {
    globalRegistry[def.Name] = &def
}

func GetSchemas() []JSONSchema {
    // Called by Agent Loop before each LLM API call
    // Returns all tool schemas so the model knows what's available
    var schemas []JSONSchema
    for _, def := range globalRegistry {
        schemas = append(schemas, def.Schema)
    }
    return schemas
}
// tools/web_search.go — self-registers at import time
func init() {
    Register(ToolDef{
        Name:        "web_search",
        Description: "Search the web for current information. Returns top 5 results with titles and snippets.",
        Dangerous:   false,
        Schema: JSONSchema{
            Properties: map[string]Property{
                "query": {Type: "string", Description: "The search query"},
            },
            Required: []string{"query"},
        },
        Fn: webSearchFn,
    })
}

func webSearchFn(ctx context.Context, args map[string]interface{}) (string, error) {
    query := args["query"].(string)

    results, err := searchClient.Search(ctx, query)
    if err != nil {
        // IMPORTANT: return the error as a string, NOT as a Go error
        // The model needs to read this error to know what happened
        return fmt.Sprintf("Search failed: %v. Try a different query or use cached data.", err), nil
    }

    return formatResults(results), nil
}
// dispatcher.go — executes tool calls from the agent loop
type Dispatcher struct {
    approvalCallback func(toolName string, args map[string]interface{}) bool
}

func (d *Dispatcher) Execute(ctx context.Context, calls []ToolCall) (Message, error) {
    var results []ToolResult

    // Run tool calls — parallel when safe, sequential when order matters
    var wg sync.WaitGroup
    resultChan := make(chan ToolResult, len(calls))

    for _, call := range calls {
        def, ok := globalRegistry[call.Name]
        if !ok {
            results = append(results, ToolResult{
                ToolUseID: call.ID,
                Content:   fmt.Sprintf("Error: tool '%s' not found in registry", call.Name),
                IsError:   true,
            })
            continue
        }

        // Approval gate for dangerous tools
        if def.Dangerous && !d.approvalCallback.Approve(call.Name, call.Arguments) {
            results = append(results, ToolResult{
                ToolUseID: call.ID,
                Content:   "User declined to execute this tool.",
            })
            continue
        }

        // Execute with timeout
        wg.Add(1)
        go func(c ToolCall, d *ToolDef) {
            defer wg.Done()
            toolCtx, cancel := context.WithTimeout(ctx, 30*time.Second)
            defer cancel()

            output, _ := d.Fn.Execute(toolCtx, c.Arguments)
            resultChan <- ToolResult{ToolUseID: c.ID, Content: output}
        }(call, def)
    }

    wg.Wait()
    close(resultChan)
    for r := range resultChan { results = append(results, r) }

    return Message{Role: "user", Content: results}, nil
}

Error Strings, Not Error Values

This is the most counterintuitive thing about tool design. When a tool fails, you should return the error as a string result, not as a Go/Python error return value. Why? Because the model needs to read the error. If you propagate it as a Go error up through the stack, the model never sees it and can't adapt its plan. If you return "Search failed: rate limited. Try again in 60 seconds or use cached data." as the tool result, the model can read that and pivot strategy.

Hermes has 61 tools built on this exact pattern. Your OpenClaw uses the same approach via the plugin system — each plugin registers its tools and the dispatcher calls them. The only real difference between a 3-tool agent and a 61-tool agent is the size of the registry. The dispatch mechanism is identical.

Evaluating Tool Dispatch

Working Well
Tool schemas are clear enough that the model calls them correctly on the first try (no retry loops due to argument confusion). Tool results are concise but complete — not too long (wastes context) nor too short (model can't use them). Parallel tool calls execute concurrently. Errors return as human-readable strings. Dangerous tools require approval and never auto-execute.
Breaking Down
Model consistently calls tools with wrong argument names (schema description is bad). Tool results are raw JSON dumps the model struggles to parse. Tools time out silently. All tool calls run sequentially even when they're independent. A tool with side effects (sending email, writing files) executes without any approval prompt.

    "name": "write_file",
    "description": "Write to a file",  // too vague

        "path": {"type": "string"},      // no description
        "data": {"type": "string"}       // model doesn't know if relative or absolute
    }
}

// Model calls it as:

PART 5

Persistence — The Diary

What It Is

Persistence is the system that stores what happened so the next session can pick up where this one left off. Without Persistence, every session starts blank — the LLM's statelessness becomes permanent. Persistence turns a stateless LLM into something that feels stateful. It's the difference between a magic 8-ball and a colleague who knows you.

Persistence is a ship's captain's log. Every day, the captain records what happened — position, weather, decisions made, course corrections, noteworthy events. The next captain who takes over the ship reads the log and immediately understands the situation. They don't have to rediscover everything from scratch. The log IS the institutional memory. Without it, every captain starts with no context about where the ship has been or why it's on its current heading.

The Two Kinds of Memory

Persistence has two fundamentally different jobs:

TypeWhatFormatRead When
Conversation HistoryThe literal messages exchanged in each sessionSQLite rows or JSON filesSession start (hydration)
Long-term MemoryDistilled facts, preferences, skills about the user/worldMarkdown files or structured DBSystem prompt assembly

OpenClaw uses files for both (your vault). Hermes uses SQLite + FTS5 for conversation history and markdown files for long-term memory (MEMORY.md, USER.md). Neither approach is objectively better — files win on debuggability, SQLite wins on query power and atomicity.

type PersistenceLayer struct {
    db        *sql.DB     // conversation history
    vaultPath string     // long-term memory files
}

// ── CONVERSATION HISTORY ─────────────────────────────────────────

func (p *PersistenceLayer) SaveTurn(sessionKey string, msg Message) error {
    tokenCount := estimateTokens(msg.Content)
    _, err := p.db.Exec(`
        INSERT INTO messages (session_key, role, content, token_count, created_at)
        VALUES (?, ?, ?, ?, ?)
    `, sessionKey, msg.Role, msg.Content, tokenCount, time.Now().Unix())
    return err
}

func (p *PersistenceLayer) LoadHistory(sessionKey string, limit int) ([]Message, error) {
    rows, err := p.db.Query(`
        SELECT role, content FROM messages
        WHERE session_key = ?
        ORDER BY created_at DESC
        LIMIT ?
    `, sessionKey, limit)
    if err != nil { return nil, err }

    var msgs []Message
    for rows.Next() {
        var m Message
        rows.Scan(&m.Role, &m.Content)
        msgs = append(msgs, m)
    }
    // Reverse to chronological
    for i, j := 0, len(msgs)-1; i < j; i, j = i+1, j-1 {
        msgs[i], msgs[j] = msgs[j], msgs[i]
    }
    return msgs, nil
}

Evaluating Persistence

Working Well
Agent references facts from past sessions accurately and without prompting. Write operations are atomic — partial writes don't corrupt history. Read latency is under 50ms. Token counts are tracked per message. Memory files are structured and human-readable. The system survives a crash — incomplete sessions don't cause corruption.
Breaking Down
Agent forgets things said 2+ sessions ago. A crash mid-session corrupts conversation history. Read times are slow (>500ms) on long sessions. Memory grows without bound (no archiving strategy). Two concurrent sessions write to the same file simultaneously and corrupt it (race condition).

Individual Failure Example: Race Condition on Concurrent Sessions

✗ BAD — Two cron jobs write to same memory file simultaneously

// Both read memory.md, append their findings, write back.
// Whoever writes last wins. The other's changes are lost.

                                               // Job B reads: "fact1\n"
    content = append(content, fact...)        // Job A: "fact1\nfact-A"
                                               // Job B: "fact1\nfact-B"

}                                              // fact-A is lost forever
✓ FIXED — File lock + atomic rename pattern
import "github.com/gofrs/flock"

func appendToMemory(path string, fact string) error {
    lock := flock.New(path + ".lock")

    // Acquire exclusive lock (blocks until available)
    locked, err := lock.TryLockContext(context.Background(), 5*time.Second)
    if err != nil || !locked {
        return fmt.Errorf("could not acquire lock: %v", err)
    }
    defer lock.Unlock()

    // Read → modify → write to temp file → atomic rename
    content, _ := os.ReadFile(path)
    content = append(content, []byte("\n- "+fact)...)

    tmpPath := path + ".tmp"
    if err := os.WriteFile(tmpPath, content, 0644); err != nil {
        return err
    }
    // Atomic rename — on Linux this is guaranteed atomic
    return os.Rename(tmpPath, path)
}

Overlap Zones — Where Parts Blur

The 5-part model is a conceptual frame, not a rigid boundary. In real systems, responsibilities bleed at the edges. These are the three main overlap zones:

OVERLAP 1: Session Hydration ↔ Persistence

The blur: Session Hydration loads data from Persistence. They're tightly coupled. In many codebases, they're the same class. The distinction is conceptual — Hydration is the "read + assemble" operation, Persistence is the "write + store" operation. Keep them logically separate even if they share a DB handle.

The risk: If Hydration also writes (e.g., "mark session as started"), you create a situation where a session is marked open but the agent loop never runs. Writes should go through Persistence only, after the loop completes.

OVERLAP 2: Agent Loop ↔ Tool Dispatch

The blur: The agent loop decides what tool to call. Tool Dispatch executes it. But who decides what tools are available? The agent loop sends tool schemas to the LLM. Where do those schemas come from? The registry, which is owned by Tool Dispatch. This creates a coupling — the loop needs to ask Dispatch what's available before it can call the LLM.

The risk: Putting tool execution logic inside the agent loop function (common in quick prototypes). This makes testing impossible — you can't test the agent loop without mocking tool execution.

OVERLAP 3: Platform Input ↔ Session Hydration

The blur: Platform Input provides the ChannelID that Session Hydration uses to look up history. But should Platform Input know about sessions? In some systems, the adapter itself checks "does this channel have an active session?" before routing the message. This is authorization bleeding into session management.

The risk: Platform-specific logic (Discord's rate limits, Telegram's retry behavior) leaking into session logic. Keep the adapter dumb — it converts events and checks allowlists, nothing more.

Poor Coordination — A Full System Failure

Here is a realistic scenario where all 5 parts are individually functional but poorly coordinated, causing a cascading failure:

✗ SCENARIO — "Dex, write my daily market summary and post it to #feed"

What should happen: Dex searches for market data, writes a summary, posts to #feed channel. Simple 3-step task.

What actually happens:

  1. [Platform Input] receives the Discord message. It also fires a second event because Discord resent the message (no dedup). Two copies of the same message enter the pipeline.
  2. [Session Hydration] runs twice, both load the same session history. Both create HydratedSession objects. Race to the agent loop.
  3. [Agent Loop] runs twice in parallel (no per-channel mutex). Both instances call web_search simultaneously.
  4. [Tool Dispatch] executes both. Both calls succeed. Both agent loops get market data. Both decide to write the summary.
  5. [Persistence] two concurrent writes to market-summary.md. One corrupts the other. File ends up with interleaved content from both writes.
  6. Both agent loops then call discord_send to post to #feed. Two identical market summaries are posted.
  7. [Persistence] both loops try to save their conversation history. Second write corrupts first. History now has duplicate entries out of order.

✓ FIXED — Coordination added at each seam
type Gateway struct {
    channelLocks *sync.Map  // one mutex per channel
    seenMessages *sync.Map  // deduplication
}

func (g *Gateway) handleMessage(msg IncomingMessage) {
    // ① Platform Input: deduplication
    msgKey := msg.Platform + ":" + msg.MessageID
    if _, exists := g.seenMessages.LoadOrStore(msgKey, time.Now()); exists {
        return  // duplicate — drop it
    }

    // ② Per-channel mutex — serialize all messages for the same channel
    lockVal, _ := g.channelLocks.LoadOrStore(msg.ChannelID, &sync.Mutex{})
    mu := lockVal.(*sync.Mutex)
    mu.Lock()
    defer mu.Unlock()

    // Now only ONE goroutine per channel runs at a time
    go func() {
        // ③ Session Hydration — now safe, no race
        session, err := hydrator.Hydrate(msg)
        if err != nil { log.Error(err); return }

        // Show typing indicator — user knows something is happening
        platform.SendTyping(msg.ChannelID)

        // ④ Agent Loop — single execution per channel
        result, err := agentLoop.Run(ctx, session)
        if err != nil {
            platform.Send(msg.ChannelID, "Sorry, I hit an error: "+err.Error())
            return
        }

        // ⑤ Persistence — atomic write, then respond
        if err := persistence.SaveResult(msg.ChannelID, session.Messages, result); err != nil {
            log.Error("persistence failed:", err)
            // Still send response — don't block user because of storage error
        }

        // Send response only after persistence succeeds (or we logged the failure)
        platform.Send(msg.ChannelID, result.Text)
    }()
}

// Key design decisions:
// 1. Dedup before ANY work is done (Input layer)
// 2. Per-channel mutex ensures serialized execution (not global lock)
// 3. Persist THEN respond — if save fails, we still send the response
// 4. Log persistence failures without blocking — user experience > storage consistency
// 5. Each concern handled by the right layer, in the right order
